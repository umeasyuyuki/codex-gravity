# Engineering Rules

このドキュメントは、実装時に必ず守る共通ルールを定義する。

## Rule 1: 疎結合実装（マイクロサービス設計思想）

新規実装・機能追加時は、以下を満たす。

1. サービス境界を先に決める
- 各機能の責務を明確にし、1サービス1責務を基本とする。

2. 契約で連携する
- サービス間連携は API 契約（入力/出力/エラー）経由に限定する。
- 内部実装への直接依存を禁止する。

3. データ所有者を明確にする
- データの正本（source of truth）を1つに定める。
- 他サービスは参照または契約経由で同期する。

4. 変更影響を局所化する
- 1つの機能変更で複数サービス内部を同時に変更しない。
- 同時変更が必要な場合は、理由と段階移行計画を記録する。

## Rule 2: 変更レポート必須（Markdown）

初回実装・機能追加・バグ修正・リファクタ・設定変更など、コード変更が発生したら必ずレポートを更新する。

1. 出力先
- `docs/reports/{task_id}.md`

2. 運用
- ファイルが無ければ `docs/reports/TEMPLATE.md` から作成する。
- 既存ファイルがある場合は同一ファイルに追記する（新規乱立しない）。

3. 必須記載項目
- 何を実装・修正したのか
- なぜその技術スタックにしたのか（非エンジニア向け）
- 駆け出しエンジニアが必ず確認すべきポイント
- 参考記事（Qiita / Zenn など）

## Rule 3: 実行ゲート

Codex Gate 実行前に以下を満たす。

1. `docs/reports/{task_id}.md` が存在する
2. 必須見出しが含まれる
3. 参考URLが1件以上ある
4. `bash .agent/skills/codex-system/scripts/validate_report.sh <report_path>` が成功する
